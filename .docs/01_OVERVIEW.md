## 한눈에 보기

**Video Trimmer**는 웹 브라우저에서 동영상의 원하는 구간만 잘라 내는 앱이다.

핵심 특징 두 가지:

- **로컬 파일은 서버에 올리지 않는다.** 내 컴퓨터의 영상은 브라우저 안에서 바로 자른다. 파일이 인터넷을 떠나지 않으니 빠르고 안전하다.
- **URL 영상은 필요한 구간만 받는다.** 유튜브·치지직 같은 온라인 영상은, 통째로 받지 않고 편집을 먼저 한 뒤 확정한 구간만 서버가 받아 준다.

기술 기반: **Next.js 16**(React 웹 프레임워크) 위에서 돌아가고, 무거운 다운로드 작업은 직접 만든 **커스텀 Node 서버**(`server.ts`)가 처리한다.

## 이 앱이 하는 일

영상을 넣는 방법이 두 가지다. 이 둘은 처리 방식이 완전히 다르다.

```
[ 입력 1: 내 컴퓨터 파일 ]  ──▶  브라우저가 직접 자름 (서버 안 거침)
[ 입력 2: 유튜브/치지직 URL ]  ──▶  스트리밍으로 편집 → 확정 구간만 서버가 받음
```

- **로컬 파일**: 드래그해서 놓으면(드래그 앤 드롭) 바로 편집 화면으로 간다. 지원 형식은 14종(MP4, WebM 등).
- **URL 영상**: 주소를 붙여 넣으면, 서버가 영상 정보를 읽어 와서 **미리보기로 재생**해 준다. 사용자는 이 미리보기 위에서 자를 구간을 정하고, 다 정한 다음에만 그 구간을 실제로 내려받는다.

편집 화면에서는 재생 막대(타임라인)를 드래그하고, 오디오 파형을 보고, 확대/축소(줌 1x~10x)하며 시작점(In)과 끝점(Out)을 정한다.

## 큰 그림: 레고 블록처럼 쌓은 구조

이 앱의 코드는 **레고 블록을 층층이 쌓은 것**처럼 정리돼 있다. 아래층은 위층을 모르고, 위층만 아래층을 가져다 쓴다. 이렇게 하면 한 곳을 고쳐도 엉뚱한 데가 안 망가진다.

이 방식을 정식 이름으로는 **FSD(Feature-Sliced Design)**라고 부른다. "기능(feature) 단위로 코드를 잘라 놓는다"는 뜻이다.

```
  app        (맨 위: 화면 주소와 API 입구)
   │  가져다 씀
   ▼
  widgets    (조각들을 합쳐 큰 화면을 만듦)
   │
   ▼
  features   (기능 하나하나: 업로드, 재생, 타임라인 …)
   │
   ▼
  shared / stores / lib   (맨 아래: 누구나 쓰는 공용 부품)
```

각 층이 맡은 일:

| 폴더 | 비유 | 하는 일 |
|------|------|--------|
| `src/app/` | 건물 정문 | 화면 주소(App Router)와 서버 API 입구(`api/`) |
| `src/widgets/` | 조립 라인 | 여러 기능을 합쳐 큰 화면을 만듦. 예: `WorkArea`가 단계별로 화면을 바꾸고, `EditingSection`이 재생기와 타임라인을 붙임 |
| `src/features/` | 기능 부서 | 기능별 폴더 5개: `upload`(업로드), `url-input`(URL 입력), `player`(재생), `timeline`(타임라인), `export`(자르기·내보내기) |
| `src/shared/` | 공용 창고 | 어디서나 쓰는 부품: 버튼·모달 같은 UI, 순수 계산 함수, 재생기 공유 상자 |
| `src/stores/` | 공용 화이트보드 | 앱 전체 상태 저장 (아래 섹션 참고) |
| `src/lib/` | 서버 뒷방 | **서버에서만** 도는 코드 (다운로드·yt-dlp·ffmpeg 등). 브라우저에서 절대 import 하면 안 됨 |

각 `features/` 폴더는 `components/`(화면), `hooks/`(로직), `utils/`(도구 함수)로 나뉜다. 필요하면 `context/`도 둔다.

### 부서끼리 직접 안 엮이게 하는 법

문제: `player`(재생) 기능과 `timeline`(타임라인) 기능은 **같은 영상 재생기**를 써야 한다. 그렇다고 한 기능이 다른 기능을 직접 가져다 쓰면, 둘이 딱 붙어 버려서 나중에 떼기 힘들다.

해결: 공용으로 쓸 물건은 **아래층 공용 창고(shared)로 내려보낸다**. 재생기 인스턴스는 `src/shared/video-player/VideoPlayerContext.tsx`에 두고, 두 기능이 여기서 꺼내 쓴다. 서로를 직접 모른 채 같은 재생기를 공유하는 것이다.

## 상태 관리: 앱의 공용 화이트보드

앱 여기저기서 "지금 재생 중인가?", "자를 시작점이 몇 초인가?" 같은 정보를 함께 봐야 한다. 이걸 컴포넌트마다 따로 들고 있으면 서로 어긋난다.

그래서 **하나의 공용 화이트보드**에 다 적어 둔다. 이 화이트보드가 **Zustand 스토어**(`src/stores/useStore.ts`)다. 모든 화면이 이 한 곳을 보고, 이 한 곳만 고친다.

화이트보드에 적힌 칸(상태 슬라이스, `src/types/store.ts`):

| 칸 | 담는 내용 |
|----|----------|
| `phase` | 지금 작업 단계 (아래 섹션) |
| `videoFile` | 영상 정보 (파일인지 URL인지, 길이, 썸네일 등) |
| `timeline` | 시작점(In)·끝점(Out)·재생 위치·줌 배율·잠금 여부 |
| `player` | 재생 중인지, 현재 시각, 볼륨, 드래그 중인지(`isScrubbing`) |
| `processing` | 업로드·자르기·다운로드 진행률 |
| `error` / `export` | 오류 정보 / 최종 결과 파일 |

### 화이트보드를 안전하게 읽고 쓰는 두 가지 규칙

**규칙 1 — 잘못된 값은 애초에 못 적게 막는다.** 끝점이 시작점보다 앞에 오면 안 된다. 이런 검사를 `src/stores/constraintUtils.ts`(`constrainInPoint`/`constrainOutPoint`/`constrainPlayhead`/`constrainZoom`)에 모아 뒀다. 값을 적을 때마다 자동으로 교정된다.

**규칙 2 — 읽는 방법이 두 가지다.**

- **화면 그릴 때**: `src/stores/hooks/` 안의 훅(`useTrimPoints`, `usePhase` 등)으로만 읽는다. 값이 바뀌면 화면이 저절로 다시 그려진다. (컴포넌트가 `useStore`를 직접 부르는 건 금지 — 이게 명시적 규칙이다.) 내부적으로 `selectorFactory.ts`의 `createStateSelector`가 `useShallow`로 감싸, 안 바뀐 값 때문에 괜히 다시 그리는 걸 막는다.
- **이벤트 처리할 때**: 마우스 드래그 같은 순간에는 "지금 이 순간의 값"이 필요하다. 이럴 땐 `src/stores/snapshot.ts`의 함수(`getStoreActions`, `getMediaSnapshot` 등)로 **즉석에서** 읽는다. 옛날 값을 붙들고 있는 실수(stale closure)를 피하려는 것이다.

정리하면: **그림 그릴 땐 훅, 순간 반응할 땐 스냅샷.** 이 구분이 이 앱의 중요한 설계 규칙이다.

### 다 쓴 물건은 스스로 치우게

영상을 새로 열거나 초기화(`reset()`)하면 이전 것을 정리해야 한다. 브라우저 메모리에 만든 임시 주소(blob URL)를 반납하고, 서버 임시 파일도 지운다(`DELETE /api/download/[jobId]`).

이때 스토어가 각 기능을 일일이 알 필요 없이, **각 모듈이 "나 치울 거 있어요" 하고 미리 등록**해 둔다(`src/lib/cleanup.ts`의 청소 목록, 정식으로는 IoC 방식). 예를 들어 FFmpeg는 로드될 때 자기 청소 함수를 등록하고, `reset()`은 등록된 청소들을 한꺼번에 실행한다(`runAllCleanups()`).

## 작업 단계(Phase): 정해진 순서로만 진행

앱은 아무 때나 아무 화면으로 튀지 않는다. **정해진 단계**를 순서대로 밟는다. 신호등이 초록→노랑→빨강 순으로만 바뀌는 것과 같다.

```
 idle ──▶ uploading ──▶ editing ──▶ processing ──▶ completed
 (대기)    (불러오는 중)   (편집 중)    (자르는 중)      (완료)
   ▲                                                    │
   └───────────────────  error  ◀───────────────────────┘
                        (문제 생기면 여기로)
```

- 단계 값은 화이트보드의 `phase` 칸에 있다.
- 단계를 바꾸는 건 복합 동작 함수가 맡는다: 문제가 생기면 `setErrorAndTransition`(→ error), 다 끝나면 `setExportResultAndComplete`(→ completed).
- `src/widgets/WorkArea.tsx`가 이 `phase` 값을 보고 **그 단계에 맞는 화면만** 보여 준다. 편집 화면처럼 무거운 부분은 그 단계가 됐을 때 비로소 불러온다(지연 로딩, 코드 스플리팅).

## 로컬 파일 자르기: 갈림길에서 길 고르기

내 컴퓨터 영상은 브라우저 안에서 바로 자른다. 그런데 영상 형식마다 자르는 도구가 다르다. 그래서 **갈림길에서 형식을 보고 길을 골라 준다**. 이 안내원이 디스패처(`src/features/export/utils/trimVideoDispatcher.ts`의 `trimVideo()`)다.

```
                자를 영상
                   │
        형식 확인 (formatDetector.ts)
        ┌──────────┼───────────────┐
     MP4 계열     그 외 형식       URL 소스
    (mov, m4v)   (webm, avi …)   (파일 아님)
        │            │               │
     MP4Box.js   FFmpeg.wasm      서버로 넘김
   (재인코딩      (브라우저용       (아래 섹션)
    없이 빠름)     ffmpeg, 정밀)
```

- **MP4 계열(ISOBMFF)** → `trimVideoMP4Box.ts`. **다시 인코딩하지 않고** 필요한 조각만 골라 붙인다(스트림 카피). 그래서 아주 빠르다. 대신 자를 수 있는 위치가 키프레임 단위라 ±1~2초 오차가 있다.
- **그 외 형식** → `trimVideoFFmpeg.ts`. 브라우저에서 도는 ffmpeg(FFmpeg.wasm, 약 20MB)를 **필요할 때만** 불러온다(`FFmpegSingleton.ts`). 느리지만 정밀하다(±0.02초).
- **URL 소스**면 여기서 바로 서버 경로(`trimVideoServer.ts`)로 넘긴다.

전체 자르기 과정은 `src/features/export/hooks/useExportState.ts`의 `handleExport`가 지휘한다.

## URL 영상: 통째로 안 받고 필요한 부분만

유튜브 영상 1시간짜리에서 10초만 필요한데, 1시간을 다 받으면 낭비다. 이 앱은 **딱 필요한 구간만** 받는다. 도서관에서 책 한 권을 통째로 빌리는 대신, 필요한 몇 쪽만 복사해 오는 셈이다.

과정은 크게 2단계다: **① 미리보기용으로 정보 읽기 → ② 확정한 구간만 받기.**

### 1단계 — 미리보기 준비 (resolve)

`src/app/api/video/resolve/route.ts`가 `yt-dlp`(영상 정보 추출 도구)로 영상 정보를 읽는다. 그리고 **영상 전체를 받지 않고도 미리보기가 재생되게** 준비한다.

- **유튜브/일반 영상 (DASH 방식)**: 영상은 화질 조각들로 나뉘어 있다. 서버는 각 조각의 **앞부분 128KB만** 살짝 받아(HTTP Range), 조각들의 위치 지도(init/sidx 구간)를 파악한다. 이 지도로 재생 안내서(MPD 매니페스트, `src/lib/dashManifest.ts`의 `buildMpd`)를 만들어, 브라우저가 필요한 조각만 그때그때 받아 재생하게 한다.
  - 이 안내서는 `manifestStore`에 담아 두고 같은 출처의 주소(`src/app/api/video/manifest/route.ts`)로 내려 준다. (다른 출처 주소면 재생기가 조각 위치를 못 찾기 때문이다.)
- **그 외 (합쳐진 영상/HLS)**: 위 방법이 안 되면 가장 좋은 단일 스트림을 골라 폴백한다.
- 한 번 읽은 정보는 잠깐 저장해 둔다(`resolveCache`).

### 2단계 — 확정 구간 다운로드

사용자가 구간을 정하고 "받기"를 누르면:

**어느 도구로 받을지 먼저 고른다** (`src/lib/platformDetector.ts`). 주소를 보고 판단한다:

- 치지직(chzzk) → **streamlink**
- 그 외(유튜브·일반) → **yt-dlp**

이렇게 상황 따라 도구를 바꾸는 방식을 전략 패턴(Strategy)이라 한다.

**yt-dlp 쪽은 두 단계로 시도한다** (`src/lib/downloadJob.ts`가 지휘, 진행 중인 작업은 `globalThis.__vtDownloadJobs`에 기록):

```
(A) 먼저: 구간 바이트만 콕 집어 받기 (byteRangeDownloader.ts)
      └─ 조각 위치표(sidx)를 읽어 필요한 바이트 범위만 받고,
         ffmpeg로 정확히 잘라 붙임. 짧은 클립에 최적.
         │  (구간이 너무 크거나 원본의 50% 이상이면 포기하고 ↓)
(B) 그 다음: 통째로 빨리 받고 로컬에서 자르기 (downloadFullThenCut)
      └─ aria2c(여러 연결로 동시에 받는 도구)로 빠르게 전체를 받고,
         내 서버에서 ffmpeg로 잘라냄 (재인코딩 없이 = 스트림 카피).
```

> 참고: 예전에는 yt-dlp의 `--download-sections` 옵션을 썼지만, 이 방식은 연결 하나로 느리게 받아서 폐기했다. 지금은 위 (A)/(B) 방식이다.

**치지직 쪽은** `src/lib/streamlinkDownloader.ts`가 처리하며, 2단계로 나뉜다: 먼저 구간을 받고, 그다음 ffmpeg로 시간 표시를 0부터 다시 맞춘다(타임스탬프 리셋, `src/app/api/video/trim/route.ts`).

**파형·스펙트럼**(`api/video/waveform`, `api/video/spectrogram`): 오디오만 뽑아(영상 안 받고) 소리 그래프를 그린다. URL을 읽는 순간 미리 받아 두고 브라우저에 캐시한다.

### 진행 상황은 실시간으로 (SSE)

다운로드가 얼마나 됐는지 실시간으로 보여 줘야 한다. 이때 **서버가 브라우저에게 한 방향으로 계속 알려 주는** 방식(SSE, Server-Sent Events)을 쓴다.

`src/features/export/utils/streamDownloadController.ts`가 이 흐름을 맡는다. 화면이 바뀌는 중에도 연결이 끊기지 않도록 **모듈 하나가 계속 붙들고** 있다(모듈 싱글턴).

```
POST /api/download/start ──▶ 작업 번호(jobId) 받음
EventSource(/api/download/stream/:jobId) ──▶ 진행률 실시간 수신
완료되면 outputUrl = /api/download/:jobId
브라우저는 이 주소에서 바로 디스크로 저장 (메모리에 안 담음)
```

진행률 계산은 `sseProgressUtils.ts`에 있다: 받는 중 0~90%, 마무리 처리 90~100%.

## 커스텀 서버: 큰 짐은 옆문으로

Next.js에는 API를 만드는 기본 통로가 있다. 그런데 **아주 크거나 오래 이어지는 데이터**(다운로드 파일, 실시간 진행률, 영상 프록시)를 이 기본 통로로 보내면, Next 내부에서 데이터를 변환하다가 메모리가 터질 수 있다(next-swc 마샬링 OOM 위험).

그래서 이런 무거운 짐은 **정문을 안 거치고 옆문으로** 곧장 보낸다. 이 옆문이 커스텀 Node 서버(`server.ts`, 빌드하면 `server.cjs`)의 **raw bypass 처리기 3종**이다:

| 옆문 | 하는 일 |
|------|--------|
| `tryServeDownload` | 완성된 파일을 그대로 흘려 보냄. 이어받기(HTTP Range 206) 지원, 주소에 UUID 검사로 경로 조작 차단 |
| `tryServeProgressSse` | 진행률을 raw SSE로 직접 내보냄 |
| `tryServeProxy` | 영상/HLS를 대신 받아 전달(프록시). m3u8 안의 주소를 고쳐 쓰고, 필요한 헤더(UA/Referer)를 붙임 |

두 통로(옆문 서버와 Next API)가 **같은 작업 목록을 공유**해야 하는데, 이걸 `globalThis`라는 전역 공간으로 이어 준다(`src/lib/downloadJob.ts`). 옆문이 처리 못 하면 Next 기본 API가 대신 받는다(폴백 병존).

## 재생 막대와 타임라인이 안 싸우게 하기

문제 상황: 사용자가 재생 위치 막대(플레이헤드)를 드래그하는 **동시에**, 영상 재생기는 "지금 3.2초 재생 중" 같은 신호(`timeupdate`)를 계속 쏜다. 둘 다 "현재 위치"를 고치려 들면 막대가 덜덜 떨린다(race condition).

해결: **사용자가 만지는 동안엔 재생기 신호를 무시한다.**

```
재생기가 timeupdate 신호를 보냄:
  IF 사용자가 드래그 중(isScrubbing) 이거나 재생기가 위치 이동 중(seeking):
      무시한다   ← 핵심! 사용자 조작을 안 덮어씀
  ELSE:
      현재 위치를 화이트보드에 기록 + 끝점(Out) 도달 시 자동 정지
```

관련 코드:

- `src/features/player/utils/playerStoreSync.ts`(`bindPlayerStoreSync`)가 재생기 신호를 스토어로 연결한다. 위 무시 규칙과 끝점 자동 정지(`playbackGuards.ts`)가 여기 있다.
- `src/shared/video-player/usePlayheadSeek.ts`(`performSeek`)는 위치 이동을 시킨 뒤 **정말 그 위치에 도착했는지**(0.1초 오차 내) 확인하고, 안 되면 타임아웃으로 되돌린다.
- `src/features/timeline/hooks/usePreviewPlayback.ts`는 스트리밍 영상 미리보기용이다. 위치를 옮긴 뒤 그 부분이 **충분히 버퍼링될 때까지 기다렸다가** 재생한다.

재생기 인스턴스는 일부러 ref가 아닌 state로 들고 있어서, 진짜 재생기가 준비되면 이를 쓰는 화면들이 다시 그려진다.

## 성능과 한계

**로컬 자르기 속도** (재인코딩 없는 경우):

| 파일 크기 | MP4Box | FFmpeg.wasm |
|-----------|--------|-------------|
| 100MB | ~1초 | ~3초 |
| 500MB | ~3초 | ~10초 |
| 1GB | ~5초 | ~20초 |

**자르기 정확도**: MP4Box는 키프레임 단위라 ±1~2초, FFmpeg.wasm은 ±0.02초. 빠름과 정밀함을 맞바꾼 것이다.

**로컬 파일 크기 한계** (브라우저 메모리 때문):

| 크기 | 상태 |
|------|------|
| < 500MB | ✅ 권장 |
| 500MB~1GB | ⚠️ 경고 |
| 1~2GB | 🟠 메모리 확인 |
| 2~5GB | 🔴 위험 |
| > 5GB | ⛔ 차단 |

URL 영상은 브라우저 메모리에 안 담고 서버→디스크로 바로 가므로 이 한계와 무관하다.

**지원 형식 (입력 14종)**: MP4·WebM·OGG·MOV·M4V·AVI·WMV·MKV·FLV·TS·3GP·3G2·MPEG·MPG. 출력은 입력 형식을 그대로 유지한다(스트림 카피).

**브라우저**: Chrome/Edge 90+, Firefox 88+, Safari 14+.
